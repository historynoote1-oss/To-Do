# Todo Website — دليل التشغيل من الموبايل

موقع فيه تسجيل دخول (username + password)، وكل حساب شايف بياناته بس.
المشروع فيه مجلدين: `backend` (API + قاعدة بيانات) و `frontend` (الموقع).

---

## الخطوة 1: ارفع الكود على GitHub

1. افتح تطبيق GitHub (أو المتصفح) وسجل دخول.
2. **+** → **New repository** → اكتب اسم (مثلاً `todo-app`) → خليه **Private** → **Create repository**.
3. فك ضغط الملف اللي معاك، وارفع فولدرين `backend` و `frontend` وملف `README.md` عن طريق **Add file → Upload files**.
4. اكتب أي commit message ودوس **Commit changes**.

✅ تأكد إن فولدرين `backend` و `frontend` ظاهرين في صفحة الريبو.

---

## الخطوة 2: الباك إند + قاعدة البيانات (Railway)

1. افتح **railway.app** → **Login with GitHub**.
2. **New Project** → **Deploy from GitHub repo** → اختار الريبو `todo-app`.
3. دوس على الـ Service اللي اتعمل → **Settings** → في **Root Directory** اكتب `backend`.
4. من نفس المشروع في Railway، دوس **New** → **Database** → **PostgreSQL**.
   Railway هيربط متغير `DATABASE_URL` تلقائيًا للـ Service بتاع الباك إند — مفيش حاجة تعملها يدوي هنا.
5. ارجع للـ Service بتاع الباك إند → تبويب **Variables** → ضيف:
   - `JWT_SECRET` = أي نص عشوائي طويل (30-40 حرف/رقم عشوائي)
   - `FRONTEND_URL` = سيبها فاضية دلوقتي، هنرجعلها في الخطوة 4
6. تأكد من **Settings**:
   - Build Command: `npm run build`
   - Start Command: `npm start`
7. دوس **Deploy** واستنى لحد ما ينجح.
8. افتح تبويب **Shell** جوه الـ Service ونفّذ:
   ```
   npx prisma migrate deploy
   ```
   ده بيعمل جداول `User` و `TodoList` و `TodoItem` في القاعدة.
9. من **Settings → Networking → Generate Domain**، هتاخد رابط زي:
   ```
   todo-backend-production.up.railway.app
   ```
   احفظه، هتحتاجه في الخطوتين الجايين.

---

## الخطوة 3: الفرونت إند (Vercel)

1. افتح **vercel.com** → **Login with GitHub**.
2. **Add New...** → **Project** → اختار نفس الريبو `todo-app` → **Import**.
3. قبل الـ Deploy: دوس **Edit** بجانب **Root Directory** واختار `frontend`.
4. افتح **Environment Variables** وضيف:
   - Name: `VITE_API_URL`
   - Value: رابط الباك إند **كامل مع https://**، مثلاً:
     ```
     https://todo-backend-production.up.railway.app
     ```
5. دوس **Deploy** واستنى دقيقة/دقيقتين.
6. هتاخد رابط زي `todo-app.vercel.app` — احفظه.

---

## الخطوة 4: اربط الفرونت إند بالباك إند (CORS)

1. ارجع لـ Railway → الـ Service بتاع الباك إند → **Variables**.
2. عدّل `FRONTEND_URL` وحط فيها رابط Vercel **كامل مع https://**، مثلاً:
   ```
   https://todo-app.vercel.app
   ```
3. دوس **Deploy** تاني عشان يتحدث.

---

## الخطوة 5: التجربة

1. افتح رابط Vercel بتاعك من أي متصفح (موبايل أو كمبيوتر، عادي زي أي موقع).
2. دوس تبويب **حساب جديد**، اكتب اسم مستخدم وكلمة مرور (6 أحرف على الأقل)، ودوس **إنشاء حساب**.
3. هتدخل على طول لواجهة قائمة المهام. جرب تضيف قائمة ومهمة.
4. لو عايز تتأكد إن الفصل بين الحسابات شغال، افتح الرابط من متصفح تاني (أو نافذة تصفح خفي) واعمل حساب تاني — هتلاقيه شايف قوائم فاضية، مش قوائمك.

---

## الخطوة 6: تفعيل حساب أدمن (اختياري)

الموقع فيه **لوحة تحكم أدمن** بتعرض كل الإحصائيات وكل المستخدمين، وبتقدر تحذف حسابات منها. الحماية هنا مقصودة تكون قوية:

- مفيش أي زرار أو شاشة في الموقع نفسه بتخلي حد "يعمل نفسه أدمن" — الطريقة الوحيدة هي إنك تدخل بنفسك على السيرفر وتشغّل أمر يدوي.
- كل طلب للوحة التحكم بيتراجع من قاعدة البيانات نفسها في كل مرة، مش من التوكن، فحتى لو حد عدّل أو زوّر التوكن بتاعه، مش هيعديه.
- فيه حماية إضافية (rate limiting) بتمنع أي حد يحاول يخمّن كلمة مرور حساب تاني بمحاولات كتيرة متكررة.
- فيه دعم للتحقق بخطوتين (2FA) لحسابات الأدمن — تفاصيله في قسم "الأمان" تحت.

### إزاي تخلي حسابك أدمن
1. اعمل حسابك العادي الأول من الموقع (تسجيل دخول → حساب جديد) لو لسه معملتوش.
2. روح Railway → الـ Service بتاع الباك إند → تبويب **Shell**.
3. اكتب الأمر ده (غيّر `username` باسم المستخدم اللي عملته فعلاً):
   ```
   npm run make-admin -- username
   ```
4. لو ظهرت رسالة `✅ الحساب "..." بقى أدمن دلوقتي` يبقى تم.
5. ارجع للموقع، سجّل خروج وسجّل دخول تاني (عشان تاخد صلاحية الأدمن الجديدة)، وهتلاقي زرار **لوحة التحكم** ظاهر جنب اسمك.

## قسم الأمان — التحقق بخطوتين (2FA) لحسابات الأدمن

من لوحة التحكم → تبويب **🔐 الأمان**، أي حساب أدمن يقدر يفعّل طبقة حماية إضافية فوق الباسورد:

1. دوس **تفعيل التحقق بخطوتين**، هيظهرلك QR code.
2. امسحه بتطبيق مصادقة على موبايلك (Google Authenticator، Authy، أو أي تطبيق TOTP تاني).
3. اكتب الكود المكوّن من 6 أرقام اللي ظهر في التطبيق عشان تأكد التفعيل.
4. هتظهرلك 10 أكواد استرجاع (recovery codes) — **احفظهم في مكان آمن فورًا**، مش هيظهروا تاني. كل كود بيتستخدم مرة واحدة بس، لو فقدت جهاز المصادقة بتاعك.

بعد التفعيل، أي تسجيل دخول لحساب الأدمن ده هيحتاج خطوة تانية: كتابة الكود من التطبيق (أو كود استرجاع) بعد الباسورد مباشرة. لو عايز تلغي التحقق بخطوتين، هتحتاج تكتب باسوردك **وكود صحيح من التطبيق سوا** — عشان محدش يقدر يلغي الحماية بسرعة حتى لو قدر يوصل لجلسة مفتوحة للأدمن.

**ملاحظة:** لو الأدمن فعّل 2FA وبعدين فقد جهازه وأكواد الاسترجاع كمان، الحل الوحيد هو دخول Railway Shell وتصفير الحقول يدويًا من قاعدة البيانات (`twoFactorEnabled`, `twoFactorSecret`, `twoFactorRecoveryCodes` على الحساب بتاعه).

## قسم الأمان — الإيميل، استرجاع كلمة المرور، وتأهيل الحسابات القديمة

النظام دلوقتي مبني على أفضل ممارسات معروفة (OWASP Authentication/Password-Storage/Forgot-Password Cheat Sheets وNIST SP 800-63B):

- **تسجيل حساب جديد** بقى محتاج إيميل إجباري (بيتبعتله إيميل تأكيد)، مش اسم مستخدم وباسورد بس زي الأول.
- **الحد الأدنى لكلمة المرور بقى 10 أحرف** بدل 8، مع رفض الباسوردات الشائعة جدًا ومنع وجود اسم المستخدم/الإيميل جواها، وفحص اختياري مقابل قاعدة بيانات "Have I Been Pwned" بطريقة آمنة (k-anonymity — الباسورد نفسه مبيتبعتش لأي حد).
- **مفيش قواعد تعقيد تعسفية** (زي "لازم رمز خاص وحرف كابيتال")، لأن الأبحاث الحديثة (NIST) بتأكد إنها بتخلي الناس تختار باسوردات أضعف وأسهل توقّع، مش أقوى. الطول هو أهم عامل.
- **نسيت كلمة المرور** بقى شغال فعليًا من صفحة تسجيل الدخول، وبيبعت رابط صالح لمدة 30 دقيقة بس على الإيميل. الرد بيفضل واحد سواء الحساب موجود أو لأ (عشان محدش يقدر "يجرب" إيميلات ويكتشف مين مسجّل عندنا).

### إعداد إرسال الإيميلات (مطلوب عشان استرجاع كلمة المرور يشتغل فعليًا)

1. اعمل حساب مجاني على [resend.com](https://resend.com) واتأكد من دومين بتاعك (أو استخدم الدومين التجريبي بتاعهم للتجربة السريعة).
2. هات API Key من لوحة Resend.
3. في Railway → الـ Service بتاع الباك إند → Variables، ضيف:
   - `RESEND_API_KEY` = المفتاح بتاعك
   - `EMAIL_FROM` = مثلاً `قائمة المهام <noreply@yourdomain.com>`
4. من غير الخطوة دي، الموقع هيفضل شغال عادي، لكن إيميلات استرجاع الباسورد وتأكيد الإيميل هتتطبع في لوجز Railway بس مش هتوصل فعليًا للمستخدم.

### تأهيل الحسابات القديمة (Username + Password بسيط، من غير إيميل)

أي حساب كان موجود قبل التحديث ده معندوش إيميل، فالنظام بيتعرّف عليه تلقائيًا كحساب "قديم". بعد أول Deploy للتحديث ده، شغّل الأمر ده **مرة واحدة بس** من Railway Shell:

```
npm run migrate-legacy-accounts
```

اللي بيحصل بعدها:
1. كل الحسابات القديمة (اللي معندهاش إيميل) بتتعلّم كـ "محتاجة إعادة تأهيل"، وبيتم تسجيل خروجهم فورًا من أي جلسة مفتوحة.
2. أول ما صاحب الحساب يجرب يسجل دخول بيوزره وباسوردة **القديمين نفسهم**، هيتوجه تلقائيًا لشاشة "تأمين الحساب" بدل الموقع العادي.
3. لحد ما يضيف إيميل وكلمة مرور جديدة قوية (10 أحرف على الأقل)، هو **ممنوع تمامًا** من استخدام أي حاجة تانية في الموقع — مفيش وصول لقوائمه ولا مهامه.
4. أول ما يخلّص الخطوة، بيدخل على طول لحسابه بكل قوائمه ومهامه القديمة **زي ما هي بالظبط من غير أي فقدان** (لأن العملية بتحدّث نفس الحساب، مش بتعمل حساب جديد).
5. تقدر تتابع مين لسه محتاج يعمل الخطوة دي من لوحة التحكم → تبويب **المستخدمين**، هتلاقي شارة "محتاج إعادة تأهيل" جنب اسمهم.

لو حبيت تحدد حسابات معينة بالاسم بدل كل الحسابات اللي من غير إيميل:
```
npm run migrate-legacy-accounts -- username1 username2
```

- **الصفحة مش بتفتح خالص** → افتح رابط الباك إند لوحده في المتصفح (`https://todo-backend-production.up.railway.app`)، لازم تشوف "Todo Backend يعمل ✅". لو ظهرت رسالة تانية، المشكلة في الباك إند مش الفرونت إند.
- **رسالة "Network Error" أو الطلبات بتفشل** → غالبًا `FRONTEND_URL` في Railway مش متظبط صح أو الـ Deploy الأخير معملوش، ارجع الخطوة 4.
- **رسالة عن كلمة المرور أو اسم المستخدم** → دي رسائل واضحة بتتولد من الكود نفسه (زي "اسم المستخدم ده مستخدم بالفعل")، مش مشكلة تقنية.

## ملاحظات
- الباسورد بيتخزن مشفّر (bcrypt)، مش نص عادي، حتى لو حد شاف قاعدة البيانات مش هيقدر يقرأه.
- تسجيل الدخول شغال بـ JWT token محفوظ في المتصفح، صالح لمدة 30 يوم.
- لو غيرت الكود بعد كده، كفاية `git push` تاني على GitHub — Railway و Vercel بيعملوا Deploy تلقائي مع كل تحديث.
- أعمدة قاعدة البيانات الجديدة بتاعة الـ 2FA بتتضاف تلقائيًا مع أول deploy بعد رفع هذا التحديث (زي جدول `Update` قبل كده) — مفيش أي أمر migration يدوي مطلوب.
- نفس الكلام على أعمدة الإيميل/استرجاع كلمة المرور/تأهيل الحسابات القديمة — بتتضاف تلقائيًا، الأمر اليدوي الوحيد المطلوب هو `npm run migrate-legacy-accounts` **مرة واحدة بس** بعد أول Deploy (تفاصيله فوق في قسم "تأهيل الحسابات القديمة").
- شاشتي "كل المستخدمين" و"سجل عمليات الأدمن" بقى فيهم بحث، تصفح بالصفحات (pagination)، وزرار تصدير CSV لكل واحدة، عشان تفضلوا سريعين حتى مع عدد كبير من المستخدمين أو العمليات.
